## 二、编辑页这轮是怎么生效的

## 1. 核心思路：改成“Markdown 源编辑 + 实时预览”

不是富文本编辑器，也不是直接改 HTML。

实际链路是：

1. 从 `/api/read` 读当前文章的 `markdown`
2. 把 markdown 塞进 CodeMirror
3. 工具栏按钮只负责插入或包裹 markdown 语法
4. 右侧 `#editPreview` 用 `marked.parse(...)` 实时渲染
5. 保存时把 markdown 提交给 `/api/save`

入口函数：`editFromView()`

对应位置：

- `static/index-main.js:848`
- `static/index-main.js:864`
- `static/index-main.js:875`
- `static/index-main.js:877`
- `static/index-main.js:937`

---

## 2. 只允许从可编辑文章进入编辑态

`editFromView()` 会先检查 iframe 当前地址是不是 `/articles/*.html`。

如果不是文章页，而是：

- 工作中心
- 今日任务
- 提醒事项

就直接拦截，不允许编辑。

这就是为什么本轮改动没有把系统页一起带进编辑器。

对应代码：

- `static/index-main.js:848`
- `static/index-main.js:851`
- `static/index-main.js:854`

---

## 3. 打开编辑页时的初始化动作

进入编辑页后，依次做了这些事：

### A. 读文件内容

通过：

- `/api/read?path=...`

拿到 `data.markdown`。

位置：

- `static/index-main.js:864`
- `static/index-main.js:865`

### B. 文件名独立为单独输入框

文件名不再靠正文第一行标题推断，而是放进：

- `#editDocTitle`

并从当前文件路径取 basename 去掉 `.html` 后写入。

位置：

- `static/index-main.js:866`
- `static/index-main.js:867`

### C. 预览区先渲染一版

预览不是直接 `marked.parse(md)`，而是走统一封装：

- `renderEditPreview(name, body)`

这样可以把“文件名展示”和“正文预览”放到同一个输出里。

位置：

- `static/index-main.js:869`
- `static/index-main.js:870`
- `static/index-main.js:919`

### D. 挂载 CodeMirror 编辑器

如果 `CodeMirrorEditor` 可用，就在 `#cmEditor` 里创建编辑器。

位置：

- `static/index-main.js:872`
- `static/index-main.js:875`
- `static/index-main.js:876`

### E. 内容变动时实时刷新预览

CodeMirror 的回调里直接重渲染：

- `renderEditPreview(document.getElementById('editDocTitle').value, val)`

位置：

- `static/index-main.js:876`
- `static/index-main.js:878`

这就是“编辑页生效”的主因：**编辑器更新和预览更新被串起来了。**

如果 `CodeMirrorEditor` 不存在，会退回到原生 `textarea`：

位置：

- `static/index-main.js:885`

这保证了即使编辑器 bundle 异常，编辑页也不至于完全不可用。

## 5. 工具栏为什么能直接生效

工具栏按钮本身都写在模板里，但真正起作用的是两个底层函数：

- `wrapMd(before, after)`
- `insertMd(text)`

位置：

- `static/index-main.js:532`
- `static/index-main.js:552`

### `wrapMd(before, after)` 的作用

对当前选区包裹 markdown 语法，比如：

- `**选中文本**`
- `*选中文本*`
- `~~选中文本~~`

对应按钮：

- 粗体
- 斜体
- 删除线
- 标题
- 列表
- 引用
- 代码块

模板调用位置：

- `templates/index.html:107`
- `templates/index.html:124`

### `insertMd(text)` 的作用

在光标位置直接插入 markdown，比如：

- `---`
- `[文字](链接)`
- `![说明](图片)`
- `[[内部链接]]`

位置：

- `static/index-main.js:552`
- `static/index-main.js:607`
- `static/index-main.js:648`
- `static/index-main.js:769`
- `static/index-main.js:831`

所以这轮的编辑能力不是“每个按钮单独写一套编辑逻辑”，而是：

- 模板按钮 → `wrapMd/insertMd` → CodeMirror/textarea → 实时预览

这是本轮能快速生效的关键。

## 6. 预览为什么比以前稳定

这一轮没有直接把正文裸渲染到预览区，而是统一走：

- `renderEditPreview(name, body)`

这个函数做了两件事：

1. 顶部显示当前文件名（sticky 样式）
2. 正文通过 `marked.parse(content)` 渲染

位置：

- `static/index-main.js:919`
- `static/index-main.js:922`
- `static/index-main.js:923`

这意味着：

- 文件名和正文解耦了
- 改标题输入框时，预览顶部同步更新
- markdown 正文不必再塞一个 H1 充当“文件名”

---

## 7. 文件名输入框为什么会联动预览

`bindTitlePreview()` 给 `#editDocTitle` 绑了 `oninput`。

每次标题变化时，会重新取正文内容，再次调用：

- `renderEditPreview(this.value, body)`

位置：

- `static/index-main.js:927`
- `static/index-main.js:930`
- `static/index-main.js:933`

所以本轮生效后，文件名区和预览头部是一套联动机制，不是保存后才刷新。

## 8. 保存链路这轮改了什么

保存入口：`saveEdit()`

位置：

- `static/index-main.js:937`

### 实际行为

1. 先读 `#editDocTitle` 作为文件名
2. 再读编辑器正文作为 markdown
3. 清理文件名非法字符
4. 新建文件走 `/api/create-file`
5. 已有文件走 `/api/save`
6. 如果改了文件名，先 `/api/rename` 再保存

这四者的完整闭环。

## 六、如果你要继续追“下划线”能力，最直接的落点在哪

按现在已经生效的结构，后续如果继续加“下划线”，改动落点会是下面这条链：

1. `templates/index.html` 的 `#mdToolbar` 增加一个按钮
2. `static/index-main.js` 里复用 `wrapMd(...)` 或新增一个专门插入下划线语法/HTML 的函数
3. 预览仍走 `renderEditPreview(...) + marked.parse(...)`
4. 保存仍走 `saveEdit()`，不需要改接口协议

- 入口：`templates/index.html:103`
