# 江月的个人笔记网站 / liruibiji

这是一个围绕“**自己长期真能用**”反复打磨出来的个人笔记网站。
## 解决了什么痛点？
1. 云端笔记库，数据在后台不属于你，分散存储，使用和分析不方便；
2. 本地笔记软件电脑和手机端同步难，随时随地记录不方便；
3. 构建知识库时AI如何接入你的笔记库。

## 设计初衷
### 极致的通用
用HTML文件驱动，不像markdown还得用特定软件打开渲染，几乎任何终端都有浏览器都能识别，所见即所得。除了环境和依赖，系统只有800K，你可以把他装在电脑、手机、云服务器的任何地方。需要用的时候，随时可以将articles目录下的笔记用优盘拷走或者线上发送，放入目录就能用。单独的HTML不依赖于笔记系统也可以发给任何AI帮你修改。
### 极致的简约。
Markdown 作为嵌入源码保存，每个 HTML 文件底部都保留 Markdown 源码。浏览器看到的是 HTML，编辑器拿到的是 Markdown。 HTML for humen， Markdown for AI，这种方式，你和AI都能一目了然。
### 极致的开放
超级酷炫的页面是能实现的，你可以让任何AI做成的任何HTML文件，放进articles目录。你也可以写个大概，然后丢给AI，让他美化返回个HTML文件，3D动画、炫酷特效、数据图表都可以实现，这很简单。
### 极致的克制
尽量保留知识库所需的记录和管理信息、辅助个人日常任务的核心功能，并将这些功能便利性尽量做到极致。提供的主要功能：日常笔记、 网络文章收录、内部双链、图片和附件上传和管理、历史版本、回收站，还有每日任务、全站笔记中的待办事项、提醒事项。笔记与系统任务可以互相联动，你不需要把所有任务都写在“工作中心”页面里。普通笔记里只要按规范写待办 / 提醒标记，系统就能自动读取出来，显示到工作中心。因为真实使用中，任务本来就应该长在上下文里。
### 提供必要的安全性和便利性
如设置网站密码、提供分享链接、提供标题模糊以便于演示时隐藏敏感信息、提供手机、电脑端和云端的安装指引。

它不是传统意义上的博客，也不是只管存文章的静态站，而是把下面几件事真正揉到了一起：

- 日常笔记
- 任务管理
- 提醒事项
- 文章收录
- 内部双链
- 附件管理
- 历史版本
- 回收站
- 移动端可用

整个项目最核心的设计思路不是“功能越多越好”，而是：

> **尽量保持内容简单、入口明确、编辑顺手、系统文件安全、普通人长期维护成本低。**

## 一、项目定位

这个项目更适合这样的使用方式：

- 用文件夹管理自己的主题知识
- 用 HTML 作为最终展示与存储格式
- 用 Markdown 作为编辑语法
- 通过少量约定标记，让系统自动识别待办、提醒、任务来源、双链关系
- 把“工作中心”作为默认首页，而不是冷冰冰的文件列表

它尤其适合：

- 个人知识库
- 长期写作与资料整理
- 带任务管理的私人笔记系统
- 本地部署 / 云服务器部署

## 二、当前已经实现的核心特点

## 1）首页不是文件列表，而是“工作中心”

进入网站后，默认先看到的是 **工作中心**，而不是根目录文件清单。

工作中心保持克制，只保留最常用入口：

- 新建笔记
- 文章收录
- 今日任务
- 提醒事项

下面只保留三个高频栏目：

- 今日任务
- 其他待办
- 提醒事项

这样做的好处：

- 打开网站第一眼就是“今天要做什么”
- 不会一上来陷入文件树
- 入口够少，长期使用不乱

---

## 2）系统任务页从普通笔记中独立出来

任务相关页面已经从普通文章目录中拆出来，单独放在：

```text
system_pages/任务管理/
```

包括：

- 今日任务.html
- 提醒事项.html
- 每日任务模板.html

这样做是为了避免两个长期问题：

1. 被误删
2. 被当成普通文章随手编辑坏掉

现在这些页面依然能编辑，但它们是“系统页”，不再和普通文章混在一起。

---

## 3）普通笔记与系统任务可以互相联动

你不需要把所有任务都写在“今日任务”页面里。

普通笔记里只要按规范写待办 / 提醒标记，系统就能自动读取出来，显示到工作中心。

这意味着：

- 任务可以写在具体项目笔记里
- 提醒可以写在具体资料页里
- 工作中心只是“汇总视图”

这个设计非常适合真实使用，因为任务本来就应该长在上下文里。

---

## 4）支持内部链接与反向链接

文章中可以插入内部链接，跳到目录树里的其他页面。

同时系统会建立反向链接区域，形成类似 Obsidian 的双链体验。

这意味着：

- A 页面链接到 B 页面
- B 页面底部能看到谁引用了它

适合知识点串联、资料索引、专题整理。

---

## 5）支持图片与附件

项目根目录下有独立附件目录：

```text
fujian/
```

编辑器支持上传：

- 图片
- 文档
- 其他附件

规则：

- 单文件大小限制 5MB
- 图片上传后可直接在文章中显示
- 非图片附件上传后以链接形式插入
- 文件名支持中文显示

并且已经补上了一个重要逻辑：

> 删除文章时，相关附件状态也要同步考虑，避免“扫描无引用附件”时误删仍在使用的文件。

---

## 6）支持回收站与历史版本

删除文件不是直接物理删除，而是进入：

```text
recycle_bin/
```

并支持：

- 回收站查看
- 回收站恢复
- 回收站搜索
- 自动清理过期内容

同时每次关键写入前会生成历史快照，保存到：

```text
snapshots/
```

可用于：

- 误修改恢复
- 历史版本查看
- 回溯内容变化

## 7）移动端已专门做过适配

这个项目不是桌面优先后顺手缩一下，而是已经针对移动端反复修过很多细节：

- 顶部按钮点击区域放大
- 侧边栏抽屉化
- 编辑页移动端预览逻辑调整
- 工作中心三栏移动端布局优化
- 弹窗高度限制、内部滚动
- 避免“外层页面 + 内层页面”双滚动条混乱

你可以理解为：

> 这个项目是按“手机上也要真能用”去修过的。

## 三、最重要的数据规则

## 1）文章最终保存为 HTML

系统的持久化文件是：

```text
.html
```

不是 `.md`。

因为这个项目要保证：

- 浏览时直接能渲染
- 分享时就是现成页面
- 系统页、普通页、收录页统一格式

---

## 2）Markdown 作为嵌入源码保存

每个 HTML 文件底部都保留 Markdown 源码：

```html
<!--source-md
# 标题

正文内容
/source-md-->
```

这样同时满足两件事：

- 浏览器看到的是 HTML
- 编辑器拿到的是 Markdown

所以它本质上是：

> **HTML 展示层 + Markdown 编辑层**

## 四、普通笔记里如何写“待办事项”和“提醒事项”

这是 README 里最重要的使用约定之一。

如果你希望 AI 或系统能从普通笔记中自动识别任务，请按下面的格式写。

---

## 1）普通待办事项写法

在任意普通文章里，可以这样写：

```md
- [ ] 完成文章收录流程复核
- [ ] 回收站恢复功能复核
- [ ] 附件上传与打开测试
```

这类会被识别为普通待办。

如果你希望带日期，也可以写：

```md
- [ ] 建立内部链接关系 📅 2026-06-18
```

---

## 2）提醒事项写法

提醒事项需要加上专门标记：

```md
- [ ] 跟进服务器部署 📅 2026-06-18 ～10:30%%提醒事项%%
```

或者只写日期，不写时间：

```md
- [ ] 检查文章收录效果 📅 2026-06-18%%提醒事项%%
```

说明：

- `%%提醒事项%%` 是识别标记
- `📅 YYYY-MM-DD` 是日期
- `～HH:mm` 是时间，可选

系统会把它收集到：

- 工作中心 → 提醒事项

---

## 3）每日任务写法

如果你希望某个任务属于“今日任务”逻辑，需要写：

```md
- [ ] 督促小孩学英语 📅 2026-06-18%%每日任务%%
```

说明：

- `%%每日任务%%` 是识别标记
- 日期必须写成当天日期，才会进入工作中心的“今日任务”

例如当天是 2026-06-18，那么只有：

```md
- [ ] 某任务 📅 2026-06-18%%每日任务%%
```

才会被归入“今日任务”。

不是当天日期的，不会进今日任务栏。

---

## 4）任务完成状态写法

未完成：

```md
- [ ] 完成任务
```

已完成：

```md
- [x] 完成任务
```

系统和 AI 都会按这个状态识别。

---

## 5）推荐写法示例

一个普通项目笔记中，可以这样同时写几种任务：

```md
# 项目复盘

- [ ] 完成文章结构整理
- [ ] 建立内部链接关系 📅 2026-06-18
- [ ] 上午检查移动端显示 📅 2026-06-18 ～09:30%%提醒事项%%
- [ ] 晚上收尾复盘 📅 2026-06-18%%提醒事项%%
- [ ] 输出最终版本 📅 2026-06-18%%每日任务%%
```

这样效果是：

- 普通待办会进“其他待办”
- 带 `%%提醒事项%%` 的会进“提醒事项”
- 带 `%%每日任务%%` 且日期是当天的会进“今日任务”

## 七、界面上那些“看似小，其实很重要”的细节

这个项目一路改下来，很多价值并不在“有某个功能”，而在于这些反复修出来的细节：

- 默认进入工作中心，而不是根目录
- 今日任务、提醒事项不再混入普通文章目录
- 系统页可以编辑，但不容易被误删
- 今日任务按钮、提醒按钮、模板按钮位置固定
- 弹窗尽量顶部对齐，避免底部按钮看不见
- 搜索结果保留，不因为点开文件就自动消失
- 回收站和历史版本有独立入口
- 附件与正文统一管理，不再散落
- 内部链接不仅能跳，还能反向建立引用关系
- 移动端优先解决“能不能操作到”，不是只追求视觉缩放

这些都是长期使用时最容易让人烦躁、但也最能体现“贴心程度”的地方。

## 十二、AI 生成笔记文章的简要规范

如果后续让 AI 帮忙生成或整理笔记，建议遵守下面的简要规则：

1. **先保留事实，不要乱补**
   - 只写用户提供的信息、已确认的内容和可验证的结果。
   - 不确定的地方要标注为“待确认”。

2. **标题统一由文件名决定**
   - 文章文件名就是主标题。
   - 正文里不要再重复造一个不同的标题，避免前后不一致。

3. **正文保持简洁、分层清楚**
   - 优先使用短段落、列表、步骤。
   - 复杂内容按“说明 / 示例 / 注意事项”展开。

4. **任务和提醒按固定标记写**
   - 待办：`- [ ] 事项`
   - 提醒：`📅 日期%%提醒事项%%`
   - 今日任务：`📅 当天日期%%每日任务%%`

5. **尽量沿用原文风格**
   - 如果是工作记录，就写得朴素直接。
   - 如果是说明文，就保持清晰、少废话。

6. **不要破坏页面结构**
   - 不随意插入多余的标题层级。
   - 不改动系统页面的固定入口和特殊按钮。

7. **输出后先检查可读性**
   - 看有没有标题重复。
   - 看有没有把任务、链接、附件标记写错。
   - 看是否适合直接保存成 HTML 页面。

> 一句话：**AI 生成内容时，优先保证“可用、准确、结构稳定”，再考虑写得好看。**
